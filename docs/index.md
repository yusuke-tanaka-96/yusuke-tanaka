### FA機器総合メーカー各種WebシステムのLift&ShiftによるAWSクラウド移行

#### 1.プロジェクト概要
- FA機器総合メーカーの社内システム・顧客向けWebサービス（公式サイト含む）をオンプレミスからAWSへLift&Shift移行
- Live環境：EC2 100台（東京80台+大阪20台）、Dev/Stg環境：EC2 40台、DNS管理：262ホストゾーン
- Terraform/AnsibleによるIaC化、GitHub Actionsを用いたCI/CDパイプラインでのインフラ開発

#### 2.担当業務
- Terraformコードの実装・改修（EC2、IAM、Route Table、VPC、SG、ALB、Route53、CloudWatch、S3、Kinesis、Firehose等）
- Terraform管理外リソースの取り込み（import）
- Ansible Playbookの作成（OS別に作り分け：AL2023、Rocky、RHEL9。ミドルウェア・DBインストール、OpenSSHアップデート、swap設定等）
- 262ホストゾーンの別AWSアカウントへの移行（262個のtfstateファイルをシェルスクリプトで自動移行）
- Lambda（Python）によるCloudWatchアラームトリガーのドメインフェイルオーバー機能の改修・テスト
- ACM証明書発行自動化ツールの実装（ShellScript）
- pgpoolフェイルオーバー時の別アカウントDNSルーティング先IPを動的に書き換えるシェルの実装・本番EC2での検証
- pgpool運用手順書の改修（初見でも復旧可能なレベルで整備）
- CloudWatchダッシュボード、Grafana、Zabbixによるリソース監視・モニタリング

#### 3.技術スタック
- AWS（EC2、IAM、VPC、ALB、Route53、CloudWatch、S3、Lambda、ACM、Kinesis、Firehose）
- Terraform
- Ansible
- GitHub、GitHub Actions
- Linux（AL2023、Rocky、RHEL9）
- Python、ShellScript
- Grafana、Zabbix
- PostgreSQL、pgpool

#### 4.成果・取り組み
- **262ホストゾーン移行の自動化を提案・実行**：262個のtfstateファイルを手動ではなくシェルスクリプトで移行することを自ら提案。検証と手順書作成を入念に行い、ミスなく完遂。IAMの切り替えが複雑な環境で、安全な移行手順を確立した
- **運用自動化ツールの実装**：ACM証明書発行の自動化（ShellScript）、pgpoolフェイルオーバー時のDNSルーティング自動書き換え等、手作業をスクリプトに置き換える改善を複数実施
- **本番環境への変更を安全に遂行**：DBを載せた本番EC2への変更作業において、入念な検証環境での動作確認と手順書作成を行い、障害なく適用
- **Terraform/Ansibleによる大規模IaC開発の実務経験**：EC2 140台超・262ホストゾーンの環境をTerraform/Ansibleで管理。OS別のPlaybook作成やTerraform管理外リソースの取り込み等、実践的なIaCスキルを習得
