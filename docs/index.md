### 大手美容室チェーン 基幹システム構築

#### 1.プロジェクト概要
- 500店舗超・100社以上のオーナー企業が利用するマルチテナント型SaaS基幹システムの新規構築
- 予約・会計・顧客・在庫・スタッフ管理・分析等を統合し、DDD（ドメイン駆動設計）ベースのモジュラーモノリスとして設計（28バウンデッドコンテキスト）
- Claude Codeを活用したAI駆動開発体制。PMがAIで設計書・仕様書を高速に生成し、各メンバーがドメインオーナーとしてレビュー・精緻化する開発スタイル
- 現在は要件定義・基本設計フェーズ（2026年3月より開発フェーズ移行予定）

#### 2.担当業務
- AWSインフラアーキテクチャの設計（構成図をdraw.ioで0から作成）
- AWS運用コストの見積もり・最適化提案
- 会計ドメインのオーナーとして、仕様書・ドメインモデルのレビュー・改善
- 全ドメイン横断の認可・権限設計のオーナーシップ（IAMポリシーとの整合を含む）
- AI生成ドキュメント（インフラ設計書等）のレビュー・品質担保
- 顧客調整・チーム横断の技術窓口

#### 3.技術スタック
- AWS（ECS Fargate、Aurora PostgreSQL Multi-AZ、ElastiCache Valkey、Redshift Zero-ETL、Lambda、CloudFront、ALB、WAF、Route53、S3、SNS/SQS、KMS、Secrets Manager）
- Terraform（構築フェーズで使用予定）
- Claude Code（AI駆動開発：仕様レビュー、ドメインモデル検証）
- draw.io（インフラ構成図・ネットワーク図作成）
- GitHub

#### 4.成果・取り組み
- **AWSインフラ全体構成の設計**：Multi-AZ構成、ECS Fargate（Graviton）、Aurora PostgreSQL、Redshift Zero-ETL等を組み合わせた構成図をdraw.ioで0から作成し、非機能要件（99.9% SLA、画面表示2秒以内、会計処理3秒以内）を満たすアーキテクチャを設計
- **AWSコスト見積もり**：月額約32万円の運用コストを算出。Gravitonインスタンス採用、NAT Gateway集約、Redshift Zero-ETL活用によるETLパイプライン不要化など、コスト最適化を設計に織り込んだ
- **会計ドメインの品質担保**：69ユースケース・75以上のAPIを持つ会計ドメイン（税計算・インボイス制度対応・赤伝/黒伝・返品/返金処理）の仕様をドメインオーナーとしてレビュー・改善し、ビジネスルールとの整合性を確保
- **全ドメイン横断の認可設計**：28バウンデッドコンテキストを横断する権限体系（4階層：オーナー＞マネージャー＞スタッフ＞ゲスト）の設計をオーナーシップを持って推進
- **AI駆動開発への適応**：PMがAIで量産する設計書・仕様書に対し、ドメインオーナーとして品質・整合性の最終責任を担う役割を遂行。従来のチームリーダー型からドメインオーナー制への組織変化を経験

### FA機器総合メーカー各種WebシステムのLift&ShiftによるAWSクラウド移行

#### 1.プロジェクト概要
- FA機器総合メーカーの社内システム・顧客向けWebサービス（公式サイト含む）をオンプレミスからAWSへLift&Shift移行
- Live環境：EC2 100台（東京80台+大阪20台）、Dev/Stg環境：EC2 40台、DNS管理：262ホストゾーン
- Terraform/AnsibleによるIaC化、GitHub Actionsを用いたCI/CDパイプラインでのインフラ開発

#### 2.担当業務
- Terraformコードの実装・改修（EC2、IAM、Route Table、VPC、SG、ALB、Route53、CloudWatch、S3、Kinesis、Firehose等）
- Terraform管理外リソースの取り込み（import）
- Ansible Playbookの作成（OS別に作り分け：AL2023、Rocky、RHEL9。ミドルウェア・DBインストール、OpenSSHアップデート、swap設定等）
- 262ホストゾーンの別AWSアカウントへの移行（262個のtfstateファイルをシェルスクリプトで自動移行）
- Lambda（Python）によるCloudWatchアラームトリガーのドメインフェイルオーバー機能の改修・テスト
- ACM証明書発行自動化ツールの実装（ShellScript）
- pgpoolフェイルオーバー時の別アカウントDNSルーティング先IPを動的に書き換えるシェルの実装・本番EC2での検証
- pgpool運用手順書の改修（初見でも復旧可能なレベルで整備）
- CloudWatchダッシュボード、Grafana、Zabbixによるリソース監視・モニタリング

#### 3.技術スタック
- AWS（EC2、IAM、VPC、ALB、Route53、CloudWatch、S3、Lambda、ACM、Kinesis、Firehose）
- Terraform
- Ansible
- GitHub、GitHub Actions
- Linux（AL2023、Rocky、RHEL9）
- Python、ShellScript
- Grafana、Zabbix
- PostgreSQL、pgpool

#### 4.成果・取り組み
- **262ホストゾーン移行の自動化を提案・実行**：262個のtfstateファイルを手動ではなくシェルスクリプトで移行することを自ら提案。検証と手順書作成を入念に行い、ミスなく完遂。IAMの切り替えが複雑な環境で、安全な移行手順を確立した
- **運用自動化ツールの実装**：ACM証明書発行の自動化（ShellScript）、pgpoolフェイルオーバー時のDNSルーティング自動書き換え等、手作業をスクリプトに置き換える改善を複数実施
- **本番環境への変更を安全に遂行**：DBを載せた本番EC2への変更作業において、入念な検証環境での動作確認と手順書作成を行い、障害なく適用
- **Terraform/Ansibleによる大規模IaC開発の実務経験**：EC2 140台超・262ホストゾーンの環境をTerraform/Ansibleで管理。OS別のPlaybook作成やTerraform管理外リソースの取り込み等、実践的なIaCスキルを習得
