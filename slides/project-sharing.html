Section 01

NTTデジタル案件

エンタープライズ向け Web3 ノードサービス構築

プロジェクト概要

  • エンタープライズ向けブロックチェーンノードのマネージドサービスを新規構築
  • 事業者・開発者がAPI経由でBitcoin/EthereumのRPCエンドポイントを利用可能にする
  • 5環境（dev / test1 / test2 / prod1 / prod2）× マルチVPC構成
  • 組織約30名。インフラチームとして参画
  • 6〜7ヶ月で本番リリース完遂 → 運用引き継ぎまで担当
30名 プロジェクト規模
5環境 マルチVPC
6〜7ヶ月 リリースまで

技術スタック

AWS

VPC EC2 ECS Fargate Aurora PostgreSQL Lambda API Gateway ALB NLB S3 CloudFront Route53 WAF KMS Secrets Manager ACM / Private CA IAM CodePipeline / CodeBuild / CodeDeploy Step Functions Kinesis Firehose

IaC

Terraform（45モジュール） GitHub Actions

監視

New Relic Sysdig CloudWatch

ドキュメント・管理

Confluence draw.io Jira

開発体制・手法

チーム構成（技術レイヤー別）

  • インフラチーム
  • バックエンドチーム
  • ブロックチェーンチーム
  • テストチーム
  • 運用チーム

開発スタイル

  • インフラはTerraformで構築
  • 設計書・提案資料はConfluenceに文書化
  • ドキュメントは基本的に人力で作成

自分の役割

  • タスク配分・技術判断
  • 顧客・DevPM・他チームとの技術折衝
  • 設計論点の整理 → 資料作成 → 意思決定

技術的なポイント

  • 5環境×マルチVPCの同時並行構築（dev / test / prod を並行でリリース）
  • ゼロトラストベースのセキュリティ設計：VPC間通信の最小権限制御、Private CA/ACMによる証明書管理
  • Terraformモジュール45種類、GitHub Actions（OIDC認証）によるPlan/Apply自動化
  • New Relic + Sysdig + CloudWatchを組み合わせた監視基盤
  • メインネット / テストネットの分離を含むブロックチェーンノードの冗長構成
Section 02

美容室チェーン案件

大手美容室チェーン 基幹システム構築

プロジェクト概要

  • 500店舗超・100社以上のオーナー企業が利用するマルチテナント型SaaS基幹システムの新規構築
  • 予約・会計・顧客・在庫・スタッフ管理・分析等を統合
  • DDD（ドメイン駆動設計）ベースのモジュラーモノリス（28バウンデッドコンテキスト）
  • AI駆動開発体制：PMがAIで設計書を高速生成 → ドメインオーナーがレビュー・精緻化
  • 2025年11月〜参画。現在は要件定義・基本設計フェーズ → 2026年3月より開発フェーズへ
500+ 店舗
28 バウンデッドコンテキスト
DDD モジュラーモノリス

技術スタック

AWS

ECS Fargate（Graviton） Aurora PostgreSQL ElastiCache Valkey Redshift Zero-ETL Lambda CloudFront ALB WAF Route53 S3 SNS/SQS KMS

IaC

Terraform（構築フェーズで使用予定）

設計手法

DDD（ドメイン駆動設計） モジュラーモノリス 28バウンデッドコンテキスト

AI・ツール

Claude Code NotebookLM Gemini（会議要約） draw.io GitHub

開発体制・手法

ドメインオーナー制

  • 技術レイヤー別のチーム制ではない
  • 各ドメインを担当する人で分担
  • 自分の担当：インフラ・会計・認可
  • ドメインオーナーが品質・整合性の最終責任を持つ

AI駆動の開発フロー

  • PMがAIで設計書・仕様書を高速生成
  • ドメインオーナーがレビュー・精緻化
  • 先方とのやり取り・合意形成

担当領域の例

  • AWSインフラ全体構成の設計
  • 会計ドメインの仕様レビュー
  • 全ドメイン横断の認可・権限設計
  • AWS運用コストの見積もり
Section 03

半年で変わったこと

AI活用と開発スタイルの進化

AI活用の変化

Before — NTTデジタル

ほぼ人力の開発

  • 設計書・提案資料はConfluenceで人力作成
  • AIはCursor + ChatGPTを補助的に利用
  • Terraformコード補完が主な用途
  • 議事録も手動で作成
→
After — 美容室チェーン

AI駆動の開発

  • 設計書・仕様書をAIで生成 → レビュー
  • Claude Codeで仕様検証・ドメインモデル設計
  • Google Meet × Geminiで会議要約を自動化
  • NotebookLMにナレッジを集約

チーム構造の変化

Before — NTTデジタル

技術レイヤー別チーム制

  • インフラ / バックエンド / ブロックチェーン / テスト / 運用
  • 各チームにリーダーがいる構造
  • 担当範囲は技術領域ごとに分かれる
  • コードを書く比重が大きい
→
After — 美容室チェーン

ドメインオーナー制

  • 各ドメインの担当者で分担
  • チーム制ではなくドメイン単位の責任
  • 担当範囲が技術横断になる
  • 設計判断・レビューの比重が大きい

半年やってみて感じること

  • AIの進化で、ドキュメント作成・設計のスピード感がまるで違う
  • その分、レビューや設計判断の重要性が増している実感がある
  • チーム制 → ドメインオーナー制で、技術領域を跨いで担当するようになった
  • 会議の要約やナレッジ管理も自動化が進んで、情報の流れ方自体が変わった
半年前と比べて、開発の進め方がかなり変わってきているなと感じています

まとめ

NTTデジタル

  • Web3ノードサービスを6〜7ヶ月で本番リリース
  • 5環境×マルチVPCの大規模インフラ
  • Terraform 45モジュール

美容室チェーン

  • 500店舗超のSaaS基幹システム
  • DDD × AI駆動開発
  • ドメインオーナー制で運営

この半年の変化

  • AI活用が補助 → 主軸に
  • チーム制 → ドメインオーナー制
  • コードを書く → 設計判断・品質担保
