Section 01

NTTデジタル案件

エンタープライズ向け Web3 ノードサービス構築

プロジェクト概要

  • エンタープライズ向けブロックチェーンノードのマネージドサービスを新規構築
  • 事業者がAPI経由でBitcoin/EthereumのRPCエンドポイントを利用可能にする
    ※ RPCエンドポイント＝ブロックチェーンネットワークへリクエストを送るためのAPI接続先
  • 2025年4月〜10月の約7ヶ月間
  • 5環境（dev / test1 / test2 / prod1 / prod2）× マルチVPC構成を同時並行で構築
  • ウォーターフォールで構築 → リリース後スクラムに移行
  • 本番リリース完遂 → 運用引き継ぎまで担当

自分の役割

  • インフラチームリーダー（最大6名）としてタスク配分・技術判断
  • 顧客・DevPM・他チームとの技術折衝
  • 設計根拠をConfluenceに文書化し、論理的に提案するスタイル
  • 主要コンポーネントの構築（API Gateway / Lambda / ログ・監視基盤 など）
Section 02

美容室チェーン案件

大手美容室チェーン 基幹システム構築

プロジェクト概要

  • 数百店舗規模・約100社のオーナー企業を想定したマルチテナント型SaaS基幹システムの新規構築
  • 予約・会計・顧客・在庫・スタッフ管理・分析等を統合
  • DDD（ドメイン駆動設計）ベースのモジュラーモノリス
  • 設計書・仕様書などあらゆるドキュメントをGitHubで一元管理
  • AI駆動開発体制：AIでドキュメントを高速生成 → ドメインオーナーがレビュー・顧客と合意形成
  • 2025年10月〜参画。要件定義・基本設計フェーズ → 2026年3月より開発フェーズへ

自分の役割

  • インフラ担当しつつ、会計ドメインも担当
  • AWSインフラ全体構成の設計
  • 会計ドメインの仕様レビュー・顧客との合意形成
  • AI生成ドキュメントのレビュー・品質担保
Section 03

半年で変わったこと

AI活用と開発スタイルの進化

AI活用の変化

Before — NTTデジタル

ほぼ人力の開発

  • 設計書はConfluence、コードはGitHub → 分散管理
  • AIはCursor + ChatGPT等を個人で補助的に利用
  • コード補完やドキュメント作成が主な用途
  • 会議の記録は各自で対応（録画など）
→
After — 美容室チェーン

AI駆動の開発

  • 設計書・仕様書をAIで生成 → レビュー
  • ドキュメントをGitHubで一元管理 → Claude Codeがリポジトリ横断で活用
  • Google Meet × Geminiで会議要約を自動化
  • NotebookLMにナレッジを集約

働き方・役割の変化

Before — NTTデジタル

チーム制 × ウォーターフォール

  • 技術レイヤー別チーム制（インフラ / BE / BC / テスト / 運用）
  • ウォーターフォールで構築、リリース後スクラムへ
  • 担当範囲はインフラ領域に特化
  • コードやドキュメントを自分で書く比重が大きい
→
After — 美容室チェーン

ドメインオーナー制 × AI駆動

  • ドメインオーナー制（各ドメインの担当者で分担）
  • DDD × AI駆動の開発フロー
  • 担当範囲が技術横断（インフラ + 会計）
  • 設計判断・レビューの比重が大きい

まとめ

  • AIの進化で、ドキュメント作成・設計のスピード感がまるで違う
  • その分、レビューや設計判断の重要性が増している実感がある
  • チーム制 → ドメインオーナー制で、技術領域を跨いで担当するようになった
  • 会議の要約やナレッジ管理も自動化が進んで、情報の流れ方自体が変わった
  • 「何を作るべきか」「なぜこの設計か」を判断し、合意形成する力がより問われるようになった
「コードを書く力」から「設計を判断し、合意形成する力」へ
— エンジニアとして求められることの重心が変わってきている
